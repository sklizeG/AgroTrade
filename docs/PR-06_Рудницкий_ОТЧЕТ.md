## 1. Введение

### 1.1. Название и краткое описание продукта

**Название продукта:** «АгроКруг»

**Краткое описание:**  
«АгроКруг» — веб-платформа для прямых поставок свежих продуктов от фермеров ресторанам и другим покупателям (B2B/B2C). Фермеры публикуют кампании предзаказа, покупатели бронируют объём и оплачивают предоплату, администратор модерирует заказы и кампании. В экосистеме продукта уже есть веб-интерфейс (ПР-04), CRM EspoCRM (ПР-05) и backend на NestJS.

**Задача ПР-06:** спроектировать и реализовать ключевой бизнес-процесс **«Обработка заказа»** на BPMS-платформе, связать его с архитектурой ПР-03 (сущность Order, статусы) и ролями ПР-05 (менеджер, администратор, фермер), выполнить интеграцию **CRM → BPMS** для получения максимального балла (15).

### 1.2. Выбранная BPMS

**Camunda Platform 7 Run** (open source, локальный Docker).

**Ссылка на платформу:** https://camunda.com/platform/  
**Локальный интерфейс:** http://localhost:8080/camunda/app/  
**Документация:** https://docs.camunda.org/manual/7.21/

Bitrix24 в этой работе **не использовался** (требование методички для ПР-06).

## 2. Этап 1 — Исследование BPMS

### 2.1. Сравнительная таблица (3 BPMS × 5 критериев)

| Критерий | Camunda 7 | ELMA365 | n8n |
|----------|-----------|---------|-----|
| **Цена** | Open source, бесплатно при self-host | Freemium / trial | Self-host бесплатно |
| **Ease of Use** | Средняя; нужен BPMN и Cockpit | Высокая, no-code | Высокая для автоматизаций, слабее для user tasks |
| **AI / Low-code** | Modeler, скрипты в шлюзах | ИИ-помощник в облаке | AI-ноды, мало классического BPMN |
| **Кастомизация** | Полный BPMN, роли, REST API | Сильная в облаке | Гибкие сценарии, но не полноценный BPMS |
| **Поддержка / RU** | EN-документация, большое комьюнити | RU-сайт, академия | EN, активное сообщество |

### 2.2. Обоснование выбора

Camunda выбрана, потому что процесс нужно было **реально связать с backend** (NestJS) и EspoCRM через REST API — это закрывает бонусный этап 5. Платформа поддерживает **стандарт BPMN 2.0**, даёт Cockpit (карта процесса) и Tasklist (эквивалент канбана по заданию), разворачивается локально через Docker без облачной регистрации.

## 3. Этап 2 — Проектирование процесса с ИИ

### 3.1. Источник сценария

Сценарий взят из **ПР-03**: жизненный цикл заказа (`pending_payment` → `reserved` → `confirmed` → `partially_delivered` → `delivered` / `cancelled`) и роли **admin**, **farmer**, **buyer** (в BPMS участвуют менеджер, администратор и фермер как исполнители задач).

### 3.2. Текстовое описание процесса

| Этап | Кто | Действие |
|------|-----|----------|
| Вход | Система | В CRM создан Lead по новому заказу с сайта |
| 1 | Менеджер | Проверить заявку (данные покупателя, кампания, сумма) |
| 2 | Менеджер | Контроль оплаты, если статус `pending_payment` |
| 3 | Администратор | Подтвердить резерв объёма |
| 4 | Фермер | Подготовить продукцию к отгрузке |
| 5 | Менеджер | Организовать отгрузку / доставку |
| 6 | Менеджер | Подтвердить факт доставки |
| 7 | Администратор | Закрыть заказ |
| Выход | — | Процесс завершён |

**Участники:** менеджер, администратор, фермер (≥ 2 роли, не только инициатор).

### 3.3. BPMN-схема

Финальная схема реализована в файле `bpms/camunda/bpmn/order-processing.bpmn` и развёрнута в Camunda.

![BPMN OrderProcessing — 7 user tasks и шлюз «Оплачен?»](pr06-assets/bpmn-diagram.png)

### 3.4. Скриншоты ИИ-диалогов

![Фрагмент диалога с ИИ в Cursor IDE при проектировании BPMS](pr06-assets/ai-dialog.png)

### 3.5. Рефлексия (проектирование)

После генерации черновика BPMN с ИИ я добавил **ветку оплаты** (`pending_payment`) и распределил задачи между тремя ролями из ПР-05. ИИ предложил линейный процесс без шлюза; вручную добавил exclusive gateway по переменной `status`, чтобы процесс соответствовал реальным статусам заказа в backend (`backend/src/common/domain.ts`, константа `ORDER_STATUSES`).

## 4. Этап 3 — Реализация в BPMS

### 4.1. Описание реализации

Процесс развёрнут в **Camunda Run** через Docker Compose. BPMN-файл монтируется в контейнер и деплоится автоматически при старте. Созданы группы **manager**, **admin**, **farmer** и пользователи Tasklist скриптом `seed-identity.mjs`.

**Эквивалент канбана:** приложение **Tasklist** + **Cockpit** (подсветка текущего шага на диаграмме).

![docker-compose.yml — развёртывание Camunda Run](pr06-assets/docker-compose.png)

### 4.2. Таблица соответствия: ПР-03 → BPMS

|---|------------------------------|-------------------|-----|---------------|
| 1 | ID заказа | `orderId` | String | `CrmOrderPayload.orderId` |
| 2 | ID лида CRM | `leadId` | String | ответ EspoCRM POST Lead |
| 3 | Статус заказа | `status` | String | `ORDER_STATUSES` в domain.ts |
| 4 | Сумма | `totalAmount` | Double | поле заказа в БД |
| 5 | Объём | `volume` | Double | поле заказа в БД |
| 6 | Email покупателя | `buyerEmail` | String | User.email |

**Сценарий скринкаста:**

Для ветки `pending_payment` процесс задерживается на задаче «Контроль оплаты» — это совпадает с логикой MVP (mock-оплата). После `POST /api/payments/mock/:orderId/pay` статус можно обновить в CRM; в ПР-08 планируется автоматическая синхронизация.
